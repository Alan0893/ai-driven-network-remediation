Owners: Matan Talvi

1. Milestone Overview
What This Milestone Is About
Milestone 1 delivers a focused, minimal end-to-end flow that proves the core detection-to-agent workflow. Logs are forwarded from the edge to the agent-service via Kafka. The agent-service filter logs and proceeds to analyze only Warning/Error logs. Analyzed logs get enriched with additional context and a remediation/escalation/on-the-fly AAP is produced accordingly.
Main Goal
Demonstrate that the system can autonomously detect an issue from logs, route it through a real event pipeline, and trigger an agentic workflow that investigates the problem, retrieves relevant runbook context, reasons about a remediation, and outputs an actionable plan — all without human intervention.
Why This Milestone Matters
This iteration validates the foundational architecture: real observability (Loki + Grafana), real messaging (Kafka), and a working AI agent (LangGraph + OGX + Granite). Every future capability — multi-cluster support, complex escalation flows, production-grade UI — builds on this proven https://github.com/mtalvi/quickstart-factory.gitfoundation. Delivering this milestone gives the team and stakeholders confidence that the core technical approach works, and provides a concrete, demoable artifact for the Red Hat AI Quickstarts catalog.

3. Scope of the First Milestone
What We Want to Achieve
Log-Forwarding: Logs from the edge pods are forwarded to the hub via Kafka.
Agent-Service Filter: The agent service filters the logs for Error/Warning logs (only these log types will be proceeded to be analyzed). 
Agentic workflow: The agent service runs the LangGraph remediation workflow:
Ingests the alert event.
Investigates the issue using MCP tools (queries pod status, events, and logs via mcp-openshift and mcp-lokistack).
Retrieves relevant runbook context from the vector store (RAG).
Calls the LLM (Granite) with the full picture (alert + investigation evidence + runbook context) to analyze the issue and produce a root cause analysis.
Decides on a remediation action:
Remediation - runs a known, pre-existing AAP job template (e.g. restart-nginx) to fix the issue, or falls back to a direct OpenShift API call (e.g. rollout restart) if AAP fails.
Escalation - creates a ServiceNow incident ticket and sends a Slack alert so a human can handle it manually; the graph doesn't wait for a response.
Lightspeed - dynamically generates a new Ansible playbook on-the-fly based on the RCA output, then runs it through AAP. Unlike remediate which uses a fixed template, lightspeed creates a custom playbook tailored to the specific incident 
Send a simple notification.
Create an audit event to a separate Kafka topic with all the details.
Simple frontend / chatbot: A minimal UI or chatbot that makes the workflow visible and demoable.
Single-cluster deployment: Everything runs on one OpenShift cluster, using namespaces to simulate edge.
What Is In Scope
Real Loki + Grafana for log collection and alerting.
Real Kafka as the event backbone (not mocked).
Agent service consuming Kafka events and running the full LangGraph graph.
Agent integration with MCP tool servers (at minimum: OpenShift, LokiStack, AAP).
Agent integration with an LLM (Granite via Llama Stack or direct model URL).
RAG context retrieval from the ingested runbooks.
Agent producing a remediation plan (playbook selection, action description).
Simple notification (Slack message or log output; no complex MCP workflow).
Simple frontend or chatbot for demo purposes.
Happy path only.
Single-cluster (single namespace simulating edge).
What Is Explicitly Out of Scope
Full multi-cluster deployment or ACM integration.
Complex Slack MCP integration (processing Slack responses, threaded workflows).
Human-in-the-loop escalation flow.
Unhappy path handling (retry logic, fallback strategies, error recovery).
Full Llama Stack integration if model URL/token can be passed directly.
Full production-grade frontend.
High-scale Kafka or infrastructure tuning.
Full Ansible/AAP remediation execution (mock/demo capability is sufficient).
Multiple remediation retry logic.
Langfuse observability integration.
Assumptions and Dependencies
An OpenShift cluster is available with sufficient resources for Loki Operator, Kafka, MinIO, and the application stack.
The Loki Operator is installed on the cluster (prerequisite for LokiStack CR).
A Granite model endpoint is accessible (either via RHOAI MaaS or a remote vLLM instance).
The existing MCP tool servers work correctly as-is and do not require modifications for V1.
The ingestion pipeline and runbook corpus are sufficient for the demo scenario.

5. Expected End-to-End Flow
1. Failure occurs on the edge workload
A pod in the dark-noc-edge namespace experiences a failure (e.g., nginx OOMKilled due to memory pressure).
The pod's logs and Kubernetes events are collected by the OpenShift log forwarder (ClusterLogForwarder) and shipped to LokiStack.
2. Alert is published to Kafka
The alert arrives on the system-alerts (or noc-alerts) Kafka topic as a structured JSON message containing the alert name, severity, affected namespace/pod, and a summary of the detected issue.
4. Agent service consumes the event
The agent service's Kafka consumer picks up the message and triggers the LangGraph remediation workflow.
5. Agent runs the agentic workflow
Node
What Happens
ingest
Parses the Kafka alert payload into structured state (event type, affected pod, namespace, severity).
investigate
Calls MCP tool servers to gather evidence about the issue: mcp-openshift.get_pods (pod status, restarts), mcp-openshift.get_events (recent warnings), mcp-openshift.get_pod_logs (error output), and mcp-lokistack.search_logs / mcp-lokistack.find_error_patterns (log analysis). This produces a rich evidence set for the LLM.
context
Queries the Llama Stack vector store (noc_runbooks) with the event details and investigation findings. Retrieves relevant runbook snippets (e.g., the nginx-oomkilled runbook with remediation steps).
analyze
Sends the full picture — alert event + MCP investigation evidence + runbook context — to the Granite LLM. The LLM returns a RootCauseAnalysis: root cause ("nginx container exceeded memory limits"), confidence (e.g., 0.92), severity (high), recommended playbook (restart-nginx or patch-deployment-memory).
decide
Evaluates confidence against thresholds. With confidence 0.92 (> 0.8), routes to execute.
execute
Generates the remediation plan: selects the appropriate playbook via MCP AAP (mcp-aap.list_job_templates) and prepares the execution parameters. For V1, producing the plan is sufficient; actual playbook execution is optional.
notify
Sends a Slack message summarizing: what was detected, what the investigation found, what the root cause is, what remediation is recommended, and the current status.

6. Results are visible
The Slack notification provides a summary of the entire flow.
The chatbot/frontend shows the workflow steps and outcome.
Grafana dashboards show the original log anomaly.
6. What We Want to Demonstrate
Demo Scenario
"Engine X Memory Issue" — An nginx pod on the simulated edge cluster runs out of memory.
The demo should walk through the following narrative:
Trigger: An operator (or automated script) causes an nginx pod in dark-noc-edge to OOMKill.
Pipeline: The alert flows through Kafka to the agent service.
Investigation: The agent queries the cluster for evidence using MCP tools — pod status and events (via MCP OpenShift), log patterns (via MCP LokiStack) — and retrieves runbook context (via RAG). The LLM then analyzes all gathered evidence to determine the root cause.
Decision: The agent determines the issue is an OOM condition with high confidence and recommends increasing memory limits or restarting the pod.
Plan: The agent produces a remediation plan referencing the appropriate Ansible playbook.
Notification: A Slack message is sent summarizing the incident and recommended action.
Visibility: The frontend/chatbot shows the complete workflow for the audience.
What Success Looks Like
The audience sees an issue detected and handled end-to-end without any human clicking buttons or running commands (after the initial trigger).
The AI agent's reasoning is visible and explainable — it shows what it found, why it decided, and what it recommends.
The system uses real infrastructure (real logs, real Kafka, real LLM) — not mocked plumbing.
The demo completes in a reasonable time (under 5 minutes from trigger to notification).
What the Demo Should Prove
Aspect
What It Proves
Technical
The LangGraph agent architecture works: state flows through nodes, MCP tools are callable, LLM produces meaningful analysis, RAG retrieves relevant context.
Architectural
The event-driven pipeline (logs → Loki → Kafka → agent) is viable for autonomous NOC operations.
Functional
The system can detect a real (simulated) issue, reason about it with AI, and produce an actionable remediation plan.
Extensibility
The modular architecture (MCP servers, graph nodes, Helm chart) is designed for future expansion (multi-cluster, escalation, human-in-the-loop).
